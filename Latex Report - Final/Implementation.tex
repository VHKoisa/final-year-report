\chapter{Implementation}

This chapter describes the technical implementation details of each internship workstream.

\section{Workstream 1: Credit Decisioning Platform Modernization}

\subsection{Initial State}

The initial platform had multiple utility repositories and a manual process for compiling rule updates into \texttt{.adb} artifacts. This model created delay and introduced avoidable human dependency in the release chain.

\subsection{Repository Consolidation}

As part of modernization, utility repositories were merged into a central codebase where feasible. The consolidation reduced code scattering, simplified dependency management, and improved discoverability of rule-related components.

\subsection{Automated Rule Build Pipeline}

A CI-triggered build workflow was introduced with the following logic: rule changes are committed and pushed, the pipeline trigger starts automatic compilation, rules are compiled into deployable \texttt{.adb} artifacts, and artifacts are validated and prepared for publishing. This transformed a manual operational step into a repeatable machine-driven process.

\subsection{Artifact Publishing and Versioning}

Compiled artifacts are published to Artifactory with controlled versioning. Versioning provides historical traceability of deployed rule packages, rollback capability in case of regression, and clear mapping between code revision and artifact revision.

\section{Workstream 2: In-house Decision Stand-in Server}

\subsection{Architecture Selection}

A prototype-ready architecture was selected, using Java Spring Boot for robust service development and integration as the backend, and React with TypeScript for maintainable user-facing interfaces and internal operations tooling on the frontend.

\subsection{Operational Role}

The stand-in server is designed as a backup-first component. It can be used when third-party decision providers are unavailable, delayed, or under maintenance. This design supports business continuity while preserving a low-risk path for incremental adoption.

\subsection{Migration Strategy}

Implementation planning included a phased migration strategy: enabling passive readiness as contingency capability, executing parity testing with third-party decision outcomes, running controlled pilots in selected domains, and gradually increasing dependency on internal service as confidence grows.

\section{Workstream 3: Fraud Central Repository with Kafka}

\subsection{Problem-Driven Design}

Fraud alerts from identity theft workflows require rapid propagation across bank systems. To avoid inconsistent decisions across teams, a central repository model was designed using asynchronous event processing.

\subsection{Event Flow Implementation}

The implemented logical flow begins when a fraud-check transaction event is received by upstream systems. A Kafka producer then publishes a standardized event to a topic, which is consumed by a Spring Boot Kafka listener. The consumer service writes or updates the record in the centralized repository, and post-investigation status updates are consumed and persisted to complete the lifecycle.

\subsection{Core Design Considerations}

To ensure production suitability, the implementation design includes idempotent event handling to limit duplicate-processing effects, retry and dead-letter handling for transient failures, status lifecycle updates to maintain auditable decision history, and monitoring hooks for consumer lag and processing health.

\section{Implementation Limitations}

Despite substantial progress, the following implementation limitations remain:

\begin{itemize}
	\item End-to-end enterprise rollout requires cross-team production approvals.
	\item Stand-in service requires long-duration parity validation before critical migration.
	\item Fraud pipeline requires deeper hardening for exactly-once semantics and advanced governance.
\end{itemize}
